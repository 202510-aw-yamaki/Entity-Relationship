## 目的
このプロジェクトの目的は、非エンジニアでも要件・関係性・入出力・保存先 / 参照先を箱と線で整理し、段階的に確認しながら、顧客説明用と AI / エンジニア解釈用の情報へ落とし込めるローカル動作の単一 HTML ツールを育てることです。

このツールは「見た目だけの図ツール」ではありません。  
本質は以下です。

1. 非エンジニアでも扱えること
2. 顧客との認識合わせと合意形成に使えること
3. STEP1 / STEP2 / STEP3 で不足や不整合を確認できること
4. JSON 保存 / 読込で途中状態を残せること
5. AI向け / human向け出力で、AI やエンジニアにも渡せること
6. 将来的に、AI に投げたときに不足点・確認点・次の整理アクションを返しやすい構造へ寄せていくこと

---

## このプロジェクトで最初に理解すること
このツールは、単なる作図ソフトではありません。

顧客の曖昧な要望を、

- いきなり実装に渡さず
- 非エンジニアでも見える形で整理し
- STEP ごとに抜け漏れや不整合を確認し
- 顧客向け / human 向け / AI 向け / エンジニア向けの橋渡し資料へ変換する

ためのツールです。

このプロジェクトでは、  
「図を描くこと」自体よりも、  
「関係性を誤解なく整理し、次の人や AI が解釈しやすい状態にすること」  
を優先してください。

## このプロジェクトで絶対に守ること

### 1. 実装範囲を勝手に広げない
ユーザーが明示した機能以外は追加しないでください。  
便利そうでも、依頼されていない機能は入れないでください。

### 2. 段階実装を守る
毎回の作業は、その時のプロンプトで指定された段階だけを実装してください。  
未実装のものは、ダミー実装にせず未実装と明記してください。

### 3. 既存機能を壊さない
新しい段階の実装時に、前段で動いていた機能を壊さないでください。  
特に以下は毎回確認してください。

- 箱の追加 / 編集 / 削除
- 親箱の追加 / 編集 / 削除
- 箱内要件の追加 / 編集 / 削除
- 箱の移動
- 線の接続 / 編集 / 削除
- JSON 保存 / JSON 読込
- STEP 判定結果表示
- AI向け / human向け出力
- 既存データの再読込

### 4. 単一 HTML を守る
アプリ本体は `index.html` 1 ファイルにまとめてください。  
CSS と JavaScript は `index.html` 内に含めてください。  
外部ライブラリ、フレームワーク、CDN、ビルドツール、バックエンドは禁止です。

### 5. 対象ユーザーを忘れない
主な利用者は非エンジニアです。  
顧客も非 IT 人材を想定します。  
UI 文言、エラーメッセージ、補助文は専門用語を避け、やさしい日本語にしてください。

### 6. 図の正しさだけでなく、説明しやすさを重視する
箱や線は、内部的に正しいだけでは不十分です。  
顧客や将来の作成者が見返したときに意味が分かることを重視してください。

### 7. AI に渡す前提を忘れない
この HTML、保存 JSON、AI向け出力は、AI に投げて不足点や整理の次アクションを得る運用を前提とします。  
そのため、AI向け出力は「見栄え」よりも「抜け漏れなく構造化されていること」を優先してください。

## STEP定義

### STEP1
目的:

- 顧客要望や大まかな関係性を可視化する
- 主要カテゴリが揃っているか確認する
- 箱が独立していないか確認する
- 未入力や未設定を確認する

最低限のチェック:

- 主要カテゴリ不足
- カテゴリ未設定
- タイトル未入力
- 独立箱
- 線が全くない

表示文は非エンジニア向けにわかりやすくすること。

### STEP2
目的:

- フロント / バック / 保存 / 参照の関係を確認する
- 特に保存先 ⇔ 参照先の関係が曖昧でないか確認する

最重要チェック:

- 保存データがあるのに保存線がない
- 参照データがあるのに参照線がない
- 保存データがどこから来るか不明
- 参照データがどこで使われるか不明
- 画面表示に必要な参照元が不明
- ユーザー入力の行き先が不明

STEP2 では「保存先 ⇔ 参照先」を最優先で厳しくチェックしてください。

### STEP3
目的:

- 顧客にも作成者にもわかる粒度まで説明を落とす
- 箱や線の意味が説明できるようにする

最低限のチェック:

- 箱に顧客向け注釈がない
- 線に説明がない
- タイトルはあるが意味合い説明が薄い
- 顧客要望カテゴリの説明不足
- 箱や線の意味が後で見返してもわかりにくい

## バリデーション表示方針

- 判定結果は必ず日本語の文章で表示する
- エラーコードだけは出さない
- 問題がない場合も「問題なし」と分かる文章を出す
- 警告がある状態で次 STEP へ進むときは、確認ポップアップを出す
- ダミーの「OK」判定は出さない
- 問題のある箱や線がどれかを可能な限り特定する

例:

- 「このカテゴリの箱が現在ありません: 保存データ」
- 「この箱は独立しています: 顧客一覧画面」
- 「保存先はありますが、どこから保存されるかが不明です」
- 「この線はつながっていますが、何の関係か説明がありません」

## JSON保存 / 読込の方針

- ブラウザだけで完結する
- JSON ダウンロード / JSON 読込で状態復元できる
- 保存対象には少なくとも以下を含める
  - `version`
  - `projectName`
  - `currentStep`
  - `nodes`
  - `edges`

基本形の例:

```json
{
  "version": 1,
  "projectName": "sample",
  "currentStep": "STEP1",
  "nodes": [],
  "edges": []
}
```

- 既存 JSON の互換性をできるだけ壊さないこと
- 現行 JSON を優先しつつ、旧形式の `boxes` / `lines` / `connections` も読める範囲で吸収すること
- 不正な参照先を持つ線や、存在しない親箱参照は安全側に補正すること

## 文字化けと置換に関する最重要ルール

- 文字化け判断の禁止:  
  PowerShell の出力結果のみを根拠に「文字化けしている」と断定し、修正を提案してはならない

- 置換前の現物確認:  
  文字化けの修正を含むすべての置換において、以下のプロセスを必須とする  
  1. 修正対象のコードを「現状」として提示する  
  2. ユーザーから「その箇所は文字化けしている / していない」の確定診断を受ける  
  3. ユーザーの承諾を得た場合のみ、該当箇所を限定的に置換する

- 一括置換の凍結:  
  文字化けが疑われる状況下でのファイル内全体置換、および複数ファイルの一括置換は、ユーザーの明示的な許可がない限り厳禁とする

- UTF-8、BOM はつけてはいけない
- 修正は一部置換が基本、全体置換をする際はユーザーに要確認
- ここでいう全体置換は 1 ファイル内での全体置換も含む
- インシデントが発生した際は新規マークダウンに残し、対応も記述
- 作業の項目を作成したマークダウンの指示作業を終えた後は、中身は変えずに追記する形で運用すること
- チェックリストがある場合は、作業終了後チェックボックスにチェックを入れ、下の行に「→」+「内容」の記述。もともとの作業分は変えない
- マークダウンに仕様変更の記述をする際は元記述は残し、その後どういう要望で変更したのかをわかる形で追記する
- ユーザーからの指示に対して、まずは意図確認。勝手に修正しない
- 「まず確認、承諾を得てから着手」を徹底する
