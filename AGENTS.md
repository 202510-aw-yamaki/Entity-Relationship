## 目的
このプロジェクトの目的は、非エンジニアでも要件・関係性・入出力・保存先 / 参照先を箱と線で整理し、段階的に確認しながら、顧客説明用と AI / エンジニア解釈用の情報へ落とし込めるローカル動作の単一 HTML ツールを育てることです。

このツールは「見た目だけの図ツール」ではありません。  
本質は以下です。

1. 非エンジニアでも扱えること
2. 顧客との認識合わせと合意形成に使えること
3. STEP1 / STEP2 / STEP3 で不足や不整合を確認できること
4. JSON 保存 / 読込で途中状態を残せること
5. AI向け / human向け出力で、AI やエンジニアにも渡せること
6. 将来的に、AI に投げたときに不足点・確認点・次の整理アクションを返しやすい構造へ寄せていくこと

---

## このプロジェクトで最初に理解すること
このツールは、単なる作図ソフトではありません。

顧客の曖昧な要望を、

- いきなり実装に渡さず
- 非エンジニアでも見える形で整理し
- STEP ごとに抜け漏れや不整合を確認し
- 顧客向け / human 向け / AI 向け / エンジニア向けの橋渡し資料へ変換する

ためのツールです。

このプロジェクトでは、  
「図を描くこと」自体よりも、  
「関係性を誤解なく整理し、次の人や AI が解釈しやすい状態にすること」  
を優先してください。

## このプロジェクトで絶対に守ること

### 1. 実装範囲を勝手に広げない
ユーザーが明示した機能以外は追加しないでください。  
便利そうでも、依頼されていない機能は入れないでください。

### 2. 段階実装を守る
毎回の作業は、その時のプロンプトで指定された段階だけを実装してください。  
未実装のものは、ダミー実装にせず未実装と明記してください。

### 3. 既存機能を壊さない
新しい段階の実装時に、前段で動いていた機能を壊さないでください。  
特に以下は毎回確認してください。

- 箱の追加 / 編集 / 削除
- 親箱の追加 / 編集 / 削除
- 箱内要件の追加 / 編集 / 削除
- 箱の移動
- 線の接続 / 編集 / 削除
- JSON 保存 / JSON 読込
- STEP 判定結果表示
- AI向け / human向け出力
- 既存データの再読込

### 4. 単一 HTML を守る
アプリ本体は `index.html` 1 ファイルにまとめてください。  
CSS と JavaScript は `index.html` 内に含めてください。  
外部ライブラリ、フレームワーク、CDN、ビルドツール、バックエンドは禁止です。

### 5. 対象ユーザーを忘れない
主な利用者は非エンジニアです。  
顧客も非 IT 人材を想定します。  
UI 文言、エラーメッセージ、補助文は専門用語を避け、やさしい日本語にしてください。

### 6. 図の正しさだけでなく、説明しやすさを重視する
箱や線は、内部的に正しいだけでは不十分です。  
顧客や将来の作成者が見返したときに意味が分かることを重視してください。

### 7. AI に渡す前提を忘れない
この HTML、保存 JSON、AI向け出力は、AI に投げて不足点や整理の次アクションを得る運用を前提とします。  
そのため、AI向け出力は「見栄え」よりも「抜け漏れなく構造化されていること」を優先してください。

## 各 Markdown の役割
- `docs/SPEC.md`
  ユーザー要望の仕様固定書です。仕様として古くなった記述は更新してください。
- `docs/IMPLEMENTATION_RULES.md`
  実装時の共通ルールです。画面構成、固定カテゴリ、固定関係種別、実装上の前提を管理します。
- `docs/COMPLETED_REQUIREMENTS.md`
  達成済みの要件だけを書く文書です。未達の項目を達成したら、この文書へ追記してください。
- `docs/UNMET_REQUIREMENTS.md`
  未達の要件だけを書く文書です。未達の項目を達成したら、この文書から削除し、達成済みへ移してください。
- `docs/VALIDATION_RULES.md`
  STEP1 / STEP2 / STEP3 の判定条件と、日本語メッセージの出し方を管理する文書です。
- `docs/JSON_SCHEMA.md`
  現行の保存 / 読込 JSON 仕様を管理する文書です。AI相談JSONなど別形式はここに混ぜないでください。
- `docs/MANUAL_TEST.md`
  変更後の手動確認手順を管理する文書です。
  `REQUIREMENTS.md`
  ユーザーの要望が書かれる文書です。この文書からユーザー要望を整理し、ユーザーの承認を得て未達要件に組み入れてください。

必要以上に Markdown を増やさないでください。  
役割が既存文書と重なる場合は、新規作成ではなく既存文書への追記・更新・統合を優先してください。

## 一時ファイルの扱い

- `tmp-*`, `*-check.*`, `debug-*`, `scratch-*` のような確認用・検査用・一時ファイルを、ユーザーの明示指示なしに作成しないこと
- 確認のために一時的な切り出しが必要でも、最終的にプロジェクト内へ残さないこと
- やむを得ず一時ファイルを作る場合は、作成理由・用途・削除可否を回答に明記すること
- 本プロジェクトでは、本体成果物に関係ない補助ファイルを勝手に残さないこと

---

## 文字化けと置換に関する最重要ルール

- 文字化け判断の禁止:  
  PowerShell の出力結果のみを根拠に「文字化けしている」と断定し、修正を提案してはならない

- 置換前の現物確認:  
  文字化けの修正を含むすべての置換において、以下のプロセスを必須とする  
  1. 修正対象のコードを「現状」として提示する  
  2. ユーザーから「その箇所は文字化けしている / していない」の確定診断を受ける  
  3. ユーザーの承諾を得た場合のみ、該当箇所を限定的に置換する

- 一括置換の凍結:  
  文字化けが疑われる状況下でのファイル内全体置換、および複数ファイルの一括置換は、ユーザーの明示的な許可がない限り厳禁とする

- UTF-8、BOM はつけてはいけない
- 修正は一部置換が基本、全体置換をする際はユーザーに要確認
- ここでいう全体置換は 1 ファイル内での全体置換も含む
- インシデントが発生した際は新規マークダウンに残し、対応も記述
- 作業の項目を作成したマークダウンの指示作業を終えた後は、中身は変えずに追記する形で運用すること
- チェックリストがある場合は、作業終了後チェックボックスにチェックを入れ、下の行に「→」+「内容」の記述。もともとの作業分は変えない
- マークダウンに仕様変更の記述をする際は元記述は残し、その後どういう要望で変更したのかをわかる形で追記する
- ユーザーからの指示に対して、まずは意図確認。勝手に修正しない
- 「まず確認、承諾を得てから着手」を徹底する

## 変更前の最低確認
何か変更する前に、最低でも以下を確認してください。

- `AGENTS.md` を読む
- `index.html` をブラウザで開いて現状を確認する

---

## 変更後の最低確認
変更後は、少なくとも以下を実施してください。

- 開始方法の選択画面が表示される
- 左 / 中央 / 右の 3 カラムが崩れない
- 箱 / 親箱 / 箱内要件 / 線の基本操作ができる
- JSON 保存 / 読込ができる
- STEP1 / STEP2 / STEP3 の結果が読める
- AI向け / human向け出力が実行できる

変更内容に応じて、該当する操作も追加で確認してください。  
例:

- Undo / Redo を入れた場合は、直前操作の巻き戻しとやり直し
- ズームを入れた場合は、拡大縮小後の操作継続
- 箱リサイズを入れた場合は、線表示追従と保存 / 読込後の再現
- 親箱改善を入れた場合は、所属関係とバリデーション表示

## 禁止事項

- 指示されていない新機能追加
- フレームワーク導入
- サーバ依存
- CDN 依存
- ダミー実装

## 迷ったときの判断

迷った場合は、機能を増やすより、現在の段階の完成度を上げてください。  
曖昧な場合は、ユーザーが最後に強調した条件を優先してください。

このプロジェクトでは、  
「広げること」より「ぶれないこと」  
を優先してください。

ただし、ユーザーが明示的に方向性を更新した場合は、  
その新しい方向性を既存思想と矛盾しない形で文書に反映し、  
共通認識を先に整えてから実装へ進んでください。
