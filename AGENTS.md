## 目的
このプロジェクトの目的は、非エンジニアでも要件・関係性・入出力・保存先/参照先を箱と線で整理し、段階的に確認しながら、顧客説明用とAI/エンジニア解釈用の情報へ落とし込めるローカル動作の単一HTMLツールを作ることです。

このツールは「見た目だけの図ツール」ではありません。
最優先は以下です。

1. 非エンジニアでもわかりやすく図を作れること
2. STEP1 / STEP2 / STEP3 の確認ができること
3. 不足や不整合を日本語文章で表示できること
4. JSON保存/読込で途中状態を残せること
5. 最終的にAI向け / human向けの出力ができること

---

## このプロジェクトで絶対に守ること

### 1. 実装範囲を勝手に広げない
ユーザーが明示した機能以外は追加しないでください。
便利そうでも、依頼されていない機能は入れないでください。

### 2. 段階実装を守る
毎回の作業は、その時のプロンプトで指定された段階だけを実装してください。
未実装のものは、ダミー実装にせず未実装と明記してください。

### 3. 既存機能を壊さない
新しい段階の実装時に、前段で動いていた機能を壊さないでください。
特に以下は毎回確認してください。

- 箱の追加 / 編集 / 削除
- 箱の移動
- 線の接続 / 編集 / 削除
- JSON保存 / JSON読込
- STEP判定結果表示
- 既存データの再読込

### 4. 単一HTMLを守る
アプリ本体は `index.html` 1ファイルにまとめてください。
CSSとJavaScriptは `index.html` 内に含めてください。
外部ライブラリ、フレームワーク、CDN、ビルドツール、バックエンドは禁止です。

### 5. 対象ユーザーを忘れない
主な利用者は非エンジニアです。
顧客も非IT人材を想定します。
UI文言、エラーメッセージ、補助文は専門用語を避け、やさしい日本語にしてください。

### 重要
文字化けはパワーシェルのエンコード由来。ユーザーには文字化けなく見えており、コードも正常。
UTF-8、BOMはつけてはいけない。
修正は一部置換が基本、全体置換をする際はユーザーに要確認。
ここでいう全体置換は1ファイル内での全体置換も含む。
インシデントが発生した際は新規マークダウンに残し、対応も記述。
作業の項目を作成したマークダウンの指示作業を終えた後は、中身は変えずに追記する形で運用すること。
チェックリストがある場合は、作業終了後チェックボックスにチェックを入れ、下の行に「→」+「内容」の記述。もともとの作業分は変えない。
マークダウンに仕様変更の記述をする際は元記述は残し、その後どういう要望で変更したのかをわかる形で追記する。
ユーザーからの指示に対して、まずは意図確認。勝手に修正しない。
「まず確認、承諾を得てから着手」を徹底する。

## ゴールとなる完成像

### コアの完成像
- ローカルでブラウザから開ける
- 箱を追加できる
- 箱を動かせる
- 箱同士を線でつなげる
- 箱のカテゴリを選べる
- 箱のタイトルと意味合いを書ける
- 箱の顧客向け注釈を書ける
- 線の種類と説明を書ける
- 線の両端の向きを設定できる
- STEP1 / STEP2 / STEP3 の確認ができる
- 各STEPで不足や不整合を日本語文章で出せる
- 各STEPで警告がある場合に確認ポップアップを出せる
- JSONで保存/読込できる

### 最終出力の完成像
- `プロジェクト名_AI.md` または `プロジェクト名_AI.txt`
- `プロジェクト名_human.md` または `プロジェクト名_human.txt`

AI向けは構造化され、一意IDを持ち、機械的に読みやすいこと。
human向けは顧客や作成者が自然文で理解しやすいこと。

## STEP定義

### STEP1
目的:
- 顧客要望や大まかな関係性を可視化する
- 主要カテゴリが揃っているか確認する
- 箱が独立していないか確認する
- 未入力や未設定を確認する

最低限のチェック:
- 主要カテゴリ不足
- カテゴリ未設定
- タイトル未入力
- 独立箱
- 線が全くない

表示文は非エンジニア向けにわかりやすくすること。

### STEP2
目的:
- フロント / バック / 保存 / 参照の関係を確認する
- 特に保存先⇔参照先の関係が曖昧でないか確認する

最重要チェック:
- 保存データがあるのに保存線がない
- 参照データがあるのに参照線がない
- 保存データがどこから来るか不明
- 参照データがどこで使われるか不明
- 画面表示に必要な参照元が不明
- ユーザー入力の行き先が不明

STEP2では「保存先⇔参照先」を最優先で厳しくチェックしてください。

### STEP3
目的:
- 顧客にも作成者にもわかる粒度まで説明を落とす
- 箱や線の意味が説明できるようにする

最低限のチェック:
- 箱に顧客向け注釈がない
- 線に説明がない
- タイトルはあるが意味合い説明が薄い
- 顧客要望カテゴリの説明不足
- 箱や線の意味が後で見返してもわかりにくい

## バリデーション表示方針

- 判定結果は必ず日本語の文章で表示する
- エラーコードだけは出さない
- 問題がない場合も「問題なし」と分かる文章を出す
- 警告がある状態で次STEPへ進むときは、確認ポップアップを出す
- ダミーの「OK」判定は出さない
- 問題のある箱や線がどれかを可能な限り特定する

例:
- 「このカテゴリの箱が現在ありません: 保存データ」
- 「この箱は独立しています: 顧客一覧画面」
- 「保存先はありますが、どこから保存されるかが不明です」
- 「この線はつながっていますが、何の関係か説明がありません」

---

## JSON保存/読込の方針

- ブラウザだけで完結する
- JSONダウンロード / JSON読込で状態復元できる
- 保存対象には少なくとも以下を含める
  - 現在STEP
  - 箱一覧
  - 線一覧
  - プロジェクト名
  - 将来拡張に備えた `version`

基本形の例:

```json
{
  "version": 1,
  "projectName": "sample",
  "currentStep": "STEP1",
  "nodes": [],
  "edges": []
}
```

後続フェーズで箱内要件や親箱を入れる場合も、JSON互換性をできるだけ壊さないでください。

## 出力ファイルの方針

### AI向け
構造重視。
必ず一意IDを含めること。

最低限含めるもの:
- プロジェクト名
- 現在STEP
- 箱一覧
- 線一覧
- 関係種別
- 方向
- 顧客要望と実装関連カテゴリの対応

### human向け
自然文重視。
非IT人材が読めること。

最低限含めるもの:
- 図全体の目的
- 各箱の意味
- 各線の意味
- 入力 → 表示 → 保存 → 参照 → 出力 の流れ
- 顧客向け注釈

## 最終フェーズで扱う追加機能

以下は最初の要望に含まれているため、後続フェーズで実装対象になり得ます。
ただし、プロンプトで指示されるまでは実装しないでください。

- 箱内に箇条書き要件を持てること
- 箱内要件同士を線でつなげること
- 箱の上にさらに大きな箱を置けること（親箱・グループ箱）

これらは「追加機能」ではなく「後続フェーズ機能」です。
先走って入れないでください。

## UIルール

- 日本語UI
- 選択中の箱 / 線が明確にわかること
- 非エンジニアでも迷いにくい配置にすること
- 余計に派手にしないこと
- 色だけに依存しない識別を入れること
- 使い方の短い説明を画面内に出すこと

---

## 実装ルール

- `index.html` の中で完結
- JavaScriptは機能ごとに関数分割
- コード内コメントは日本語
- 無理に短くせず、読みやすさ優先
- ダミーボタン禁止
- コンソールエラー禁止
- グローバル状態は整理して持つ
- IDは衝突しないようにする
- 保存→読込→再判定が壊れないこと

## Codexが各タスクで返すべき内容

毎回の作業後は、回答の最後に以下を必ず書いてください。

1. 変更したファイル一覧
2. 今回実装したこと
3. 今回あえて実装していないこと
4. 手動確認手順
5. 既知の注意点

常に次の順で優先してください。

1. 指定されたSTEPの要件
2. 非エンジニアにわかりやすいこと
3. 保存/読込の安定性
4. 既存機能の維持

## 禁止事項

- 指示されていない新機能追加
- フレームワーク導入
- サーバ依存
- CDN依存
- ダミー実装

## 迷ったときの判断

迷った場合は、機能を増やさず、現在の段階の完成度を上げてください。
曖昧な場合は、ユーザーが最後に強調した条件を優先してください。

このプロジェクトでは、
「広げること」より「ぶれないこと」
を優先してください。
