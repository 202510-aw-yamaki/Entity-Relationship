- 箱追加
- 親箱追加
- 線接続開始
- 保存
- 読込
- STEP 切替
- 判定実行
- 出力
- 使い方の短い説明

### 中央
図エリア

- 箱配置
- 親箱表示
- 線表示
- 箱内要件表示
- 選択状態の視覚化
- ドラッグ移動

### 右カラム
選択中要素の詳細編集

- 箱編集
- 線編集
- 箱内要件編集
- STEP 結果表示
- 現在状態の要約

## 固定する箱カテゴリ
カテゴリは勝手に増やさず、まずは以下を固定してください。

- 保存データ
- 参照データ
- 出力結果
- 注釈

プロンプトで明示された場合のみ増減してください。

## 固定する線の関係種別
関係種別はまず以下を基本にしてください。

- 表示
- 入力
- 保存
- 参照
- 出力
- 関連

プロンプトで明示された場合のみ変更してください。

## 箱データの必須項目
箱は最低でも以下を持ってください。

- `id`
- `category`
- `title`
- `meaning`
- `customerNote`
- `x`
- `y`
- `width`
- `height`
- `isGroup`
- `parentId`
- `items`

`items` は箱内の箇条書き要件です。  
`parentId` は親箱との所属関係です。  
`isGroup` は親箱か通常箱かを区別するために使います。

## 箱内要件データの必須項目
箱内要件は最低でも以下を持ってください。

- `id`
- `title`
- `description`

## 線データの必須項目

## STEP定義

### STEP1
目的:

- 顧客要望や大まかな関係性を可視化する
- 主要カテゴリが揃っているか確認する
- 箱が独立していないか確認する
- 未入力や未設定を確認する

最低限のチェック:

- 主要カテゴリ不足
- カテゴリ未設定
- タイトル未入力
- 独立箱
- 線が全くない

表示文は非エンジニア向けにわかりやすくすること。

### STEP2
目的:

- フロント / バック / 保存 / 参照の関係を確認する
- 特に保存先 ⇔ 参照先の関係が曖昧でないか確認する

最重要チェック:

- 保存データがあるのに保存線がない
- 参照データがあるのに参照線がない
- 保存データがどこから来るか不明
- 参照データがどこで使われるか不明
- 画面表示に必要な参照元が不明
- ユーザー入力の行き先が不明

STEP2 では「保存先 ⇔ 参照先」を最優先で厳しくチェックしてください。

### STEP3
目的:

- 顧客にも作成者にもわかる粒度まで説明を落とす
- 箱や線の意味が説明できるようにする

最低限のチェック:

- 箱に顧客向け注釈がない
- 線に説明がない
- タイトルはあるが意味合い説明が薄い
- 顧客要望カテゴリの説明不足
- 箱や線の意味が後で見返してもわかりにくい

## バリデーション表示方針

- 判定結果は必ず日本語の文章で表示する
- エラーコードだけは出さない
- 問題がない場合も「問題なし」と分かる文章を出す
- 警告がある状態で次 STEP へ進むときは、確認ポップアップを出す
- ダミーの「OK」判定は出さない
- 問題のある箱や線がどれかを可能な限り特定する

例:

- 「このカテゴリの箱が現在ありません: 保存データ」
- 「この箱は独立しています: 顧客一覧画面」
- 「保存先はありますが、どこから保存されるかが不明です」
- 「この線はつながっていますが、何の関係か説明がありません」

## JSON保存 / 読込の方針

- ブラウザだけで完結する
- JSON ダウンロード / JSON 読込で状態復元できる
- 保存対象には少なくとも以下を含める
- `version`
- `projectName`
- `currentStep`
- `nodes`
- `edges`

基本形の例:

```json
{
  "version": 1,
  "projectName": "sample",
  "currentStep": "STEP1",
  "nodes": [],
  "edges": []
}
```

- 既存 JSON の互換性をできるだけ壊さないこと
- 現行 JSON を優先しつつ、旧形式の `boxes` / `lines` / `connections` も読める範囲で吸収すること
- 不正な参照先を持つ線や、存在しない親箱参照は安全側に補正すること

## 出力ファイルの方針

### AI向け
構造重視。  
必ず一意 ID を含めること。

最低限含めるもの:

- プロジェクト名
- 現在 STEP
- 箱一覧
- 親箱一覧
- 箱内要件一覧
- 線一覧
- 関係種別
- 方向
- 顧客要望と実装関連カテゴリの対応
- STEP の判定結果
- AI が追加質問や不足指摘をしやすい材料

### human向け
自然文重視。  
非 IT 人材が読めること。

最低限含めるもの:

- 図全体の目的
- 各箱の意味
- 各線の意味
- 入力 → 表示 → 保存 → 参照 → 出力 の流れ
- 顧客向け注釈

### AI出力の今後の扱い
現時点では AI向け Markdown / Text を基本とする。  
ただし、ユーザーから明示指示があった場合は、AI に投げやすい JSON 形式の設計・改善も対象とする。

## UIルール

- 日本語 UI
- 選択中の箱 / 線 / 箱内要件が明確にわかること
- 非エンジニアでも迷いにくい配置にすること
- 余計に派手にしないこと
- 色だけに依存しない識別を入れること
- 使い方の短い説明を画面内に出すこと
- STEP1 / STEP2 / STEP3 が何を確認する段階なのか分かるようにすること
- 線の始点 / 終点が分かりやすいこと
- 直接操作を増やす場合でも、既存操作より分かりにくくしないこと

---

## 実装ルール

- `index.html` の中で完結
- JavaScript は機能ごとに関数分割
- コード内コメントは日本語
- 無理に短くせず、読みやすさ優先
- ダミーボタン禁止
- コンソールエラー禁止
- グローバル状態は整理して持つ
- ID は衝突しないようにする
- 保存 → 読込 → 再判定が壊れないこと
- 既存データを読めなくする変更は避けること
- 操作追加時は、マウス操作同士の競合に注意すること
- Undo / Redo やズームなどの共通操作を入れる場合は、既存の選択・ドラッグ・接続操作を壊さないこと
